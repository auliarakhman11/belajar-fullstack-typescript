# 2.1. Teori Normalisasi & Granular Access Control (Multi-Outlet)

Dalam sistem POS skala besar, kita tidak hanya membatasi kasir, tetapi juga harus bisa mengatur agar seorang Admin hanya bisa mengelola cabang-cabang tertentu yang ditugaskan kepadanya oleh Super User.

## 1. Konsep Many-to-Many Mapping

Untuk mendukung fitur di mana satu user bisa mengakses satu atau lebih cabang secara spesifik, kita tidak bisa lagi meletakkan `outlet_id` langsung di dalam tabel `User`. Kita membutuhkan **Tabel Perantara (Pivot Table)**.

- **Tabel: User_Outlet_Assignments**
  - Kolom: `user_id`
  - Kolom: `outlet_id`
  - Kolom: `is_primary` (Boolean, untuk menandai cabang utama kasir)

## 2. Implementasi Logika Berdasarkan Role

Meskipun secara database relasinya adalah Many-to-Many, kita menerapkan **Aturan Bisnis (Business Rules)** di level aplikasi:

### A. Role: Cashier (Kasir)

- **Aturan:** Hanya boleh memiliki **tepat satu** baris di tabel `User_Outlet_Assignments`.
- **Flow:** Saat Super User membuat akun kasir, sistem hanya mengizinkan pemilihan satu outlet.
- **Scope:** Saat kasir login, sistem hanya mengambil satu ID outlet tersebut.

### B. Role: Admin & Super User

- **Aturan:** Boleh memiliki **satu atau lebih** baris di tabel `User_Outlet_Assignments`.
- **Flow:** Super User bisa mencentang cabang mana saja (Cabang A, C, dan E) yang boleh diakses oleh Admin tersebut.
- **Scope:**
  - Di Dashboard Admin, filter "Pilih Cabang" hanya akan menampilkan daftar cabang yang sudah di-assign kepadanya.
  - Admin tersebut **tidak akan melihat** opsi Cabang B atau D jika tidak diberikan akses oleh Super User.

## 3. Mengapa Menggunakan Tabel Perantara?

1. **Fleksibilitas Tinggi:** Jika suatu saat seorang kasir dipindahkan (mutasi) atau diperbantukan ke cabang lain, kita cukup mengubah data di tabel perantara tanpa mengubah data profil user.
2. **Audit & Keamanan:** Kita bisa melacak siapa saja yang punya akses ke sebuah cabang dengan sangat mudah.
3. **Pemisahan Data:** Menghindari kolom `outlet_id` yang bernilai `NULL` di tabel User, yang secara teori normalisasi database (Clean Data) jauh lebih baik.

## 4. Efek pada Laporan Penjualan

Dengan struktur ini, query untuk laporan menjadi:
`SELECT * FROM transactions WHERE outlet_id IN (SELECT outlet_id FROM user_outlet_assignments WHERE user_id = 'id_admin_terkait')`

Ini memastikan Admin hanya bisa melihat data yang menjadi haknya.
