# 1.1. Konsep Pemisahan Monolith ke API-Driven (Micro-Frontend)

Sebelum kita menulis satu baris kode pun, kita harus memahami mengapa kita memilih teknologi Next.js (Frontend) dan Express.js (Backend), bukan menggabungkannya dalam satu tempat.

Dalam pengembangan web, terdapat dua pola arsitektur utama: **Monolithic** dan **API-Driven (Decoupled)**.

## 1. Arsitektur Monolithic

Pada arsitektur tradisional (seperti saat menggunakan framework Laravel dengan view Blade, atau PHP murni dengan Bootstrap), seluruh sistem berada di satu tempat.

- **Cara kerja:** Saat user meminta halaman web, server akan mengambil data dari database, lalu merender HTML (menggabungkan data dan desain) di sisi server, baru kemudian mengirimkan halaman utuh ke browser user.
- **Kelemahan untuk POS:** Jika kita ingin membuat aplikasi mobile (Android/iOS) di masa depan, kita tidak bisa menggunakan kode tampilan (Blade/HTML) yang sama. Kita harus membangun sistem dari awal. Selain itu, sistem ini sangat bergantung pada koneksi internet yang stabil antara klien dan server untuk merender halaman.

## 2. Arsitektur API-Driven (Pilihan Kita)

Untuk proyek POS berskala Enterprise ini, kita memisahkan sistem menjadi dua "aplikasi" yang berdiri sendiri:

1. **Backend (Express.js + PostgreSQL):** Hanya bertugas mengelola logika bisnis, database, dan keamanan. Backend sama sekali tidak peduli dengan warna tombol atau bentuk tabel. Output dari backend hanyalah sekumpulan data mentah berformat **JSON**.
2. **Frontend (Next.js + Tailwind CSS):** Bertugas mengatur antarmuka (UI/UX). Frontend akan "menelpon" Backend untuk meminta data JSON, lalu memolesnya menjadi tampilan yang cantik.

### Mengapa Ini Sangat Penting untuk POS?

Sistem Point of Sale (POS) di kasir menuntut dua hal krusial:

1. **Kecepatan UI:** Kasir tidak boleh menunggu _loading_ (halaman _refresh/reload_) setiap kali menekan tombol tambah barang. Dengan Next.js, perpindahan antar menu terjadi secara instan di browser.
2. **Kemampuan Offline:** Karena Frontend adalah aplikasi mandiri yang berjalan di browser, kita bisa menggunakan **Service Worker** dan **IndexedDB** untuk mencegat aksi _save_ saat internet putus. Data akan disimpan secara lokal di komputer kasir, dan UI tetap merespons seolah-olah online. Saat internet kembali, Frontend akan diam-diam mengirim data tersebut ke Backend Express.js di latar belakang (_background sync_).

Pemisahan ini membuat pemeliharaan aplikasi jauh lebih mudah. Jika di masa depan sistem ini sukses dan Anda ingin membuat aplikasi mobile untuk owner memantau laporan, Anda cukup membuat aplikasi Flutter yang memanggil Backend (API) yang sama tanpa perlu merombak sistem yang sudah ada.
