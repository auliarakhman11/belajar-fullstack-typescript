# 1.2. Anatomi JWT: Access Token, Refresh Token, dan Keamanan Stateless

Karena Frontend (Next.js) dan Backend (Express.js) kita terpisah, kita memerlukan cara aman agar Frontend bisa membuktikan "siapa dirinya" setiap kali meminta data ke Backend. Di sinilah **JWT (JSON Web Token)** berperan.

## 1. Stateful vs Stateless Authentication

Secara tradisional, aplikasi menggunakan sistem **Stateful (Session-based)**:

- Saat user login, server membuat "Session ID" dan menyimpannya di dalam memori/database server, lalu memberikan ID tersebut ke browser.
- **Kelemahan:** Jika server di-restart, semua user akan _logout_. Jika user bertambah jutaan, memori server akan penuh karena harus mengingat siapa saja yang sedang login.

Untuk arsitektur API-Driven, kita menggunakan sistem **Stateless (Token-based)**:

- Saat user login, server memverifikasi password, lalu menerbitkan sebuah "Kartu Identitas Digital" berupa teks panjang (Token JWT) dan memberikannya ke browser. Server **tidak menyimpan** token ini di database.
- Setiap kali Frontend meminta data, ia harus melampirkan JWT ini. Server cukup memverifikasi tanda tangan kriptografi pada token tersebut untuk mengetahui apakah token itu valid atau palsu.

## 2. Anatomi JWT

Sebuah JWT sebenarnya adalah teks _Base64_ yang terdiri dari 3 bagian yang dipisahkan oleh titik (`.`):

1. **Header:** Berisi informasi algoritma enkripsi yang digunakan (misal: HS256).
2. **Payload:** Berisi data _user_ yang tidak rahasia (misal: `user_id`, `role`, dan waktu kedaluwarsa). **Jangan pernah** memasukkan password di sini karena Payload bisa di-decode oleh siapa saja.
3. **Signature:** Ini adalah kunci keamanannya. Dibuat dengan menggabungkan Header, Payload, dan sebuah **Secret Key** yang hanya diketahui oleh server (disimpan di file `.env` Backend). Jika seseorang mengubah Payload (misalnya, mengubah `role` dari 'kasir' menjadi 'admin'), Signature-nya akan tidak cocok dan server akan menolaknya.

## 3. Masalah Keamanan & Solusi: Access Token + Refresh Token

Jika kita hanya menggunakan satu JWT, kita menghadapi dilema keamanan:

- Jika masa aktifnya lama (1 bulan): User (kasir) nyaman tidak perlu sering login. Tapi jika token ini dicuri oleh _hacker_, _hacker_ punya akses penuh selama 1 bulan.
- Jika masa aktifnya sebentar (15 menit): Sangat aman. Tapi kasir akan marah karena setiap 15 menit mereka ter-logout otomatis saat sedang melayani pelanggan.

**Solusinya adalah menggunakan sistem Dua Token:**

1. **Access Token (Umur Pendek - misal 15 Menit):**
   Token ini digunakan Frontend untuk mengambil data dari Backend. Disimpan di dalam memori (variabel Frontend) sehingga aman dari serangan pencurian lewat script jahat (XSS).
2. **Refresh Token (Umur Panjang - misal 7 Hari):**
   Token ini hanya digunakan untuk satu hal: **Meminta Access Token baru ketika Access Token lama kedaluwarsa**. Token ini dikirim oleh server dan disimpan di dalam **HTTP-Only Cookie**. _Cookie_ ini sangat aman karena tidak bisa dibaca oleh JavaScript/Frontend sama sekali.

### Alur Kerja di Aplikasi POS Kita Nanti:

1. Kasir login di pagi hari. Server mengirimkan Access Token (15 menit) dan Refresh Token (7 hari di _cookie_).
2. Kasir melakukan transaksi. Frontend mengirim Access Token ke server. Transaksi sukses.
3. Setelah 16 menit, kasir melakukan transaksi lagi. Server menolak karena Access Token _expired_ (kedaluwarsa).
4. Di belakang layar (tanpa diketahui kasir), _library_ Axios di Next.js kita akan mencegat penolakan ini, secara otomatis mengirim Refresh Token ke server untuk meminta Access Token baru, lalu mengulang transaksi kasir tadi.
5. Kasir tidak merasakan apa-apa dan tetap lancar melayani pelanggan.

Inilah standar industri untuk aplikasi modern yang menjamin keamanan tinggi tanpa mengorbankan kenyamanan _user experience_ (UX).
